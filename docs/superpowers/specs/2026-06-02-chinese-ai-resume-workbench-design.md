# 中文 AI 简历工作台设计方案

日期：2026-06-02

状态：待用户审核

## 背景

这个方案是基于 Reactive Resume 做二次开发，目标是把它改造成一个中文优先的 AI 简历产品。

目标用户先聚焦在技术方向应届生、大学生、初级程序员。他们最常见的需求不是单纯“做一份好看的简历”，而是：

- 不知道怎么从零填写简历，需要 AI 引导。
- 已经有旧简历，希望上传 PDF 或 Word 后继续优化。
- 有目标岗位 JD，希望简历能针对这个岗位改得更准。

我们前面看过几个视觉和产品方向，最后确定采用 C 的精修方向：一个统一的简历工作台。中间是简历文档，左侧是不同投递目标/版本，右侧是 AI 审稿和 Patch 建议。

这个项目可以作为自己的二开项目来做。Reactive Resume 是 MIT 协议，可以二次开发和商业化，但必须保留原项目的版权、License 和必要法律声明。用户界面上可以换成自己的品牌，但不能抹掉应保留的开源归属。

## 产品目标

做一个中文优先的 AI 简历工作台，而不是简单换皮的英文简历编辑器。

第一版产品应该像一个成熟的生产力工具：

- 简历文档是主角。
- AI 是审稿助手，不是喧宾夺主的聊天机器人。
- 视觉上克制、清晰、专业，不像后台管理系统，也不像黑色程序员控制台。
- 用户能围绕真实投递场景完成简历创建、优化、版本管理和导出。

工作台里必须让这些动作非常明显：

- 编辑简历内容。
- 上传旧简历。
- 粘贴或绑定 JD。
- 查看 AI 对简历的诊断和优化建议。
- 像审代码一样逐条接受、拒绝、改写 AI Patch。
- 回退 AI 修改。
- 保存不同岗位版本。
- 导出 PDF 和 Word/DOCX。

## 现有能力

Reactive Resume 已经有不少可以直接复用的底子，不需要从零做。

已有能力包括：

- PDF 导出。
- 后端 PDF 下载接口。
- DOCX/Word 导出，来自 `@reactive-resume/docx`。
- JSON 导出和导入。
- PDF 简历 AI 解析接口。
- DOCX 简历 AI 解析接口。
- AI 聊天和简历 Patch 提案能力。
- 基于 JSON Patch 的简历局部更新能力。
- 简历版本冲突保护。
- AI Patch 应用前的快照，可以用于恢复。
- 简历复制、锁定、公开分享、模板、布局控制、实时预览。
- 已经存在 `zh-CN` 和 `zh-TW` locale 文件，但当前产品还不是中文优先。

这些能力应该被重新包装到新的中文工作台里，而不是全部重写。

## 产品形态

产品采用一个统一工作台，而不是多个割裂页面。

### 左侧：投递项目和版本

左侧不再只是普通“简历列表”，而是“投递项目”。

比如：

- 字节前端实习投递版
- 腾讯 PCG 前端投递版
- 通用校园版

每一个投递项目都可以理解为：同一份基础经历，针对某个岗位生成的一份简历版本。

### 中间：简历文档

中间区域是视觉主角，展示当前简历。

用户应该感觉自己正在处理一份真实可投递的简历，而不是在填写一个后台表单。

第一版仍然使用 Reactive Resume 原来的结构化简历数据作为底层数据，不做完整 Word 那种自由画布。这样可以保证 PDF/DOCX 导出、模板渲染、AI Patch 和数据校验都稳定。

### 右侧：AI 审稿和 Patch

右侧是 AI 能力区，主要做三件事：

- JD 匹配度分析。
- 简历问题诊断。
- 可审核的 Patch 建议。

AI 不应该直接静默修改简历。它应该给出修改建议，让用户逐条确认。

每条 Patch 都应该说明：

- 改哪里。
- 为什么改。
- 改完对目标 JD 有什么帮助。
- 这个建议是推荐应用、需要确认，还是风险较高。

### 顶部：当前状态和导出

顶部显示：

- 当前投递目标。
- 当前版本号。
- 保存状态。
- 预览。
- 保存版本。
- 导出 PDF。
- 导出 Word/DOCX。

## 编辑模型

第一版不做完整 Word 克隆。

原因是：Word 式自由编辑会破坏简历数据结构，后续 PDF、DOCX、模板、AI Patch、版本回退都会变复杂。

更合理的第一版做法是：

- 底层仍然使用 Reactive Resume 的 `ResumeData`。
- 用户看到的是一份简历文档。
- 点击简历里的某个区域时，自动聚焦到对应编辑项。
- AI Patch 可以高亮它影响的简历段落。
- 富文本内容继续使用已有富文本编辑能力。
- 支持常见快捷键，比如保存、撤销、重做、预览、导出。

DOCX 导出必须保留。用户如果想在 Word 里进一步精修，可以下载 Word 文件后继续改。

## 导入和导出

### PDF 导出

必须保留，并放到更显眼的位置。

可以复用现有的 PDF 生成能力，包括浏览器端的 `createResumePdfBlob` 和后端 PDF 下载接口。

### Word/DOCX 导出

必须保留，并在中文界面里叫“导出 Word”或“导出 DOCX”。

可以复用现有的 `buildDocx` 能力。

文案要说清楚：导出的 Word 文件可以继续在 Microsoft Word 或 Google Docs 里编辑。

### 旧简历导入

第一版需要支持三种导入：

- PDF
- DOCX/Word
- JSON

PDF 和 DOCX 导入应该调用现有 AI 解析接口，把旧简历变成结构化简历数据。

导入后，用户应该直接进入同一个工作台，而不是进入另一个完全不同的向导页面。

导入后的内容应该被视为草稿，可以马上做 AI 诊断，也可以马上粘贴 JD 优化。

## AI 工作流

AI 的定位是“简历审稿助手”，不是“自动替你乱改”。

主要流程：

1. 用户粘贴 JD，或选择一个目标岗位。
2. 系统提取岗位名称、公司、关键词、职责、能力要求。
3. 系统把 JD 和当前简历做对比。
4. 系统生成 Patch 建议，而不是直接修改简历。
5. 用户逐条查看 Patch。
6. 用户选择接受、拒绝、改写、解释。
7. 接受后的 Patch 通过 JSON Patch 更新简历。
8. 用户可以把已应用的 AI Patch 回退到应用前。

Patch 卡片需要包含：

- 标题：具体问题是什么。
- 原因：为什么这对目标岗位重要。
- 建议：具体要怎么改。
- 影响范围：影响简历哪个段落或字段。
- 状态：推荐应用、需要确认、风险较高、仅供参考。
- 操作：接受、拒绝、改写、解释、恢复。

AI 不能编造用户经历。

如果 AI 想写“性能提升 30%”这类内容，但用户没有提供真实数据，它必须标记为“需要确认”，让用户自己补真实信息。

## 回退、撤销和版本

这里有三种不同概念，不能混在一起。

### AI Patch 回退

这个能力现有代码里已经有基础。

AI Patch 应用前会保存 `snapshotData`。用户可以把简历恢复到某个 AI Patch 应用前的状态。

新工作台里应该把它显示成：

- 恢复到应用前
- 此 Patch 已回退
- 此 Patch 和后续 Patch 已一起回退
- 简历已变化，无法安全回退，需要重新生成建议

### 手动撤销和重做

这是现有产品里还需要补的能力。

第一版应该支持当前编辑会话里的撤销/重做，也就是用户手动编辑时可以按：

- Ctrl+Z 撤销
- Ctrl+Y 或 Ctrl+Shift+Z 重做

这个能力主要处理当前会话中的结构化编辑，不需要跨很多天恢复历史。

### 保存版本

保存版本是长期版本管理，不是撤销。

用户应该能显式保存：

- 通用校园版
- 字节 JD 版
- 腾讯 JD 版
- 修改前备份

建议新增一个持久化版本表，保存每个版本的快照。

建议字段：

- `resume_version.id`
- `resume_version.resumeId`
- `resume_version.userId`
- `resume_version.label`
- `resume_version.source`
- `resume_version.snapshotData`
- `resume_version.createdAt`

其中 `source` 可以是：

- manual：用户手动保存
- ai_patch：AI Patch 后保存
- import：导入后保存
- export：导出前保存

## 视觉方向

采用 C 精修方向。

整体感觉应该是：

- 简历文档居中。
- 中文优先。
- 成熟、克制、清楚。
- 少一点卡片堆叠感。
- 少一点后台管理味。
- 主要用中性色，保留一个蓝色作为主操作色。
- 成功、警告、风险状态只用少量绿色、橙色、红色。
- 通过细线、留白、字体层级来建立质感。

避免：

- 大量 SaaS 仪表盘卡片。
- 黑色控制台风格作为主界面。
- 紫色渐变。
- 装饰性光斑、圆球、花哨背景。
- 页面里出现营销落地页式 Hero。
- AI 自动改内容但不给用户审核。

Mockup 里暂用品牌名“锐历”。最终名字可以再换，但产品气质要保持中文优先。

## 技术架构

第一版应该做成一个“工作台层”，复用现有 Reactive Resume 能力。

### 前端

需要新增或改造一个中文优先的工作台路由。

它应该组合这些现有能力：

- 简历草稿数据。
- 简历预览。
- PDF 导出。
- DOCX 导出。
- 旧简历导入。
- AI Chat。
- AI 简历分析。
- AI Patch。
- AI Patch 回退。

前端需要新增：

- 投递项目侧栏。
- 更显眼的导出工具栏。
- AI Patch 审稿面板。
- 当前会话撤销/重做。
- 保存版本入口。
- 中文品牌和中文文案。

### 后端

后端优先复用：

- 简历 CRUD。
- 简历 patch。
- PDF export。
- AI parse PDF。
- AI parse DOCX。
- AI chat。
- agent action revert。

后端可能需要新增：

- 保存版本表。
- 保存版本接口。
- 恢复某个保存版本接口。
- 目标岗位/JD 数据存储。

如果目标岗位数据可以短期存在 resume metadata 里，第一版可以先不单独建复杂表；但长期建议独立出来，方便做岗位库、投递记录和多版本管理。

## 数据流

主要数据流程：

1. 用户创建、导入或打开一份简历。
2. 工作台加载简历数据和当前投递目标。
3. 用户编辑结构化简历数据。
4. 前端记录当前会话的撤销/重做历史。
5. 自动保存或显式保存调用现有简历更新接口。
6. 用户粘贴 JD 后，系统生成分析和 Patch 建议。
7. 用户接受某条 Patch。
8. Patch 通过 JSON Patch 更新简历，并保存应用前快照。
9. 用户可以回退 AI Patch。
10. 用户可以保存当前版本。
11. 用户导出 PDF 或 Word。

## 错误处理

### 导入失败

如果 PDF/DOCX 解析失败，或者 AI Provider 不可用，界面要显示中文错误。

上传状态不能直接消失，要允许用户重试。

### Patch 冲突

如果 Patch 生成后，简历已经被用户改过，要提示：

当前简历已变化，这条建议可能不再适用。

用户可以选择重新生成建议。

### 回退冲突

如果恢复 AI Patch 时发现后续内容已经变化，要提示：

无法安全恢复到应用前，请先保存当前版本，或重新生成建议。

### 导出失败

PDF 或 DOCX 生成失败时，用户应该留在工作台里，并看到明确的重试提示。

### 保存失败

自动保存失败不能丢失本地编辑内容。

界面应该区分：

- 已保存
- 保存中
- 保存失败

## 测试范围

### 前端测试

需要覆盖：

- 工作台用中文渲染。
- 左侧投递项目存在。
- 中间简历预览存在。
- 右侧 AI Patch 面板存在。
- PDF 和 DOCX 按钮调用现有导出函数。
- PDF/DOCX/JSON 导入能进入工作台草稿。
- AI Patch 卡片能显示接受、拒绝、改写、解释、恢复状态。
- 手动撤销/重做能处理代表性编辑。

### 后端测试

需要覆盖：

- 保存版本能存快照。
- 保存版本能恢复。
- AI Patch 仍然能通过现有快照回退。
- 过期 Patch 会返回版本冲突。
- 目标岗位/JD 信息不会意外出现在导出的简历正文里。

### 浏览器验证

需要实际打开本地工作台检查：

- 桌面尺寸下没有文字重叠。
- 窄屏尺寸下工具栏、侧栏、AI 面板不会挤坏。
- PDF 导出按钮能点。
- DOCX 导出按钮能点。
- 保存版本和恢复 Patch 的状态能看懂。

## 第一版范围

第一版要做：

- 中文优先工作台 UI。
- PDF 导出入口。
- Word/DOCX 导出入口。
- PDF/DOCX/JSON 导入入口。
- JD 匹配。
- AI Patch 审稿面板。
- AI Patch 回退。
- 当前会话的手动撤销/重做。
- 保存版本快照。
- 中文品牌和中文文案。

第一版不做：

- 完整 Microsoft Word 克隆。
- 任意拖拽文本框的自由排版画布。
- 付费系统。
- 团队协作。
- 企业 ATS 对接。
- 公开营销官网重做。
- 从零重写简历 schema。
- 移除 MIT 协议要求保留的法律声明。

## 验收标准

第一版完成时，用户应该可以：

- 用中文界面创建简历。
- 上传 PDF、DOCX 或 JSON 旧简历。
- 粘贴 JD。
- 查看 JD 匹配和 AI 建议。
- 逐条接受或拒绝 AI Patch。
- 回退已应用的 AI Patch。
- 保存一个命名版本。
- 导出 PDF。
- 导出 Word/DOCX。

界面应该符合 C 精修方向：

- 简历居中。
- AI 在右侧审稿。
- 左侧是投递项目。
- 整体成熟、专业、中文优先。
- 不像后台卡片堆砌。

同时：

- 现有 PDF/DOCX 导出能力不能被破坏。
- 导入和 AI Provider 错误要用中文讲清楚。
- 产品可以使用自己的品牌，但必须保留原 MIT License 和必要版权声明。
