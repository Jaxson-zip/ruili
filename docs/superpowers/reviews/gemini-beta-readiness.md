# 锐历 (Ruili) Beta 上线前 QA 与竞品分析报告

> 对标竞品：OrtonY/smart-resume

## 重点功能状态检查

- **注册登录**：已实现（基础邮箱/密码/OAuth 等），但需确认生产环境下是否有顺畅的兜底方案以防邮件推送链路失败。
- **简历创建**：已实现，基础编辑器及数据结构可用。
- **模板切换**：已实现，支持多种开源/定制模板实时预览与切换。
- **PDF/DOCX 导出**：均已实现。但 DOCX 导出会因为流式 HTML 转换丢失复杂双栏排版布局。
- **PDF/DOCX/图片导入**：已实现，导入前的 AI/OCR 配置阻断提示已补齐；仍重度依赖外部 AI/OCR Providers，解析效果需在真实 Provider 下复测。
- **AI Provider 配置**：已实现，支持用户端配置与环境变量服务端兜底。
- **OCR 配置**：已实现，目前仅支持 Azure 方案，且密钥临时储存于浏览器本地。
- **AI 单条润色**：已修复，编辑器内单条内容润色能力已落地，待结合真实 JD 与用户配置的 LLM 复测。
- **JD 派生副本**：已修复，支持从现有简历结合 JD 派生副本，待复测副本命名、数据隔离与失败提示。
- **工作台真实简历预览**：已修复，工作台不再依赖空态/占位预览，待复测多模板与长简历场景。
- **部署 env**：文档（`chinese-resume-beta.md`）基本闭环，`compose.yml` 提供了详细组件编排。

## 缺口清单与建议

### 【P0 级】严重影响核心体验或产品定位

#### 1. 单条经历的 AI 深度润色与 JD 匹配已修复，待复测
* **复测步骤**：
  1. 进入编辑器页面。
  2. 新增或编辑某一段工作经历/项目经历。
  3. 使用单条 AI 润色/扩写能力，并分别测试无 LLM 配置、用户自配 LLM、环境变量兜底 LLM 三种状态。
* **当前状态**：已修复，单条 AI 润色入口和接口能力已补齐，不再是 Beta 缺失项。
* **剩余风险**：该能力仍依赖用户或部署方配置可用的 LLM Provider；如果没有平台级免费额度，首次用户仍会遇到“需要先配置”的门槛。需要复测提示文案是否清楚、失败后是否保留原文、生成内容是否能稳定回填到正确字段。

#### 2. AI/OCR Provider 阻断提示已修复，配置门槛仍是剩余风险
* **复测步骤**：
  1. 首次注册/登录新账号。
  2. 在 Dashboard 直接点击导入，选择图片或 DOCX 文件。
  3. 在未配置 AI Provider 或 OCR Provider 时确认导入动作被明确阻断，并给出可执行的设置引导。
* **当前状态**：已修复，导入链路的 AI/OCR 阻断提示已补齐，不再表现为晦涩报错或长时间 Pending。
* **剩余风险**：体验仍不是竞品式开箱即用。LLM 需要用户或部署方配置；OCR 仍主要是 Azure Document Intelligence/浏览器本地配置方案（Storage key: `ruili.browser-ocr-provider`），跨设备漫游和国内低门槛 OCR 仍需后续迭代。

### 【P1 级】影响进阶体验和部分转化

#### 3. JD 派生副本已修复，待复测
* **复测步骤**：
  1. 在 Dashboard 拥有一份满意的“基础版简历”。
  2. 基于这份简历和某家公司的 JD，生成一份定制化副本。
  3. 检查原简历不被覆盖，新副本可打开、可预览、可继续编辑。
* **当前状态**：已修复，已支持结合 JD 派生副本，不再只能手工复制后逐行改写。
* **剩余风险**：仍需复测长 JD、空 JD、LLM 超时、生成失败、重复点击以及副本命名冲突等边界；该能力同样依赖用户或部署方配置可用 LLM。

#### 4. 小内存 VPS 部署提示已部分修复，稳定性仍需压测
* **复测步骤**：
  1. 按照 `chinese-resume-beta.md` 提及的最低配置（1核1G）启动单机 Docker 服务。
  2. 多人（或压测）同时点击导出 PDF。
  3. 确认部署文档与 Docker 运行提示均明确提醒小内存环境需要 Swap/内存预留。
* **当前状态**：已部分修复，VPS/Docker 小内存提示已补齐，可降低自托管用户误配概率。
* **剩余风险**：提示不能替代资源隔离。PDF 导出依赖无头浏览器（Puppeteer/Playwright），在 1C1G 或并发导出下仍可能 OOM、变慢或 Pending，Beta 前仍建议压测并观察重启恢复行为。

### 【P2 级】体验瑕疵，可在 Beta 后迭代

#### 5. DOCX 导出的排版保真度存在上限
* **复现步骤**：
  1. 在编辑器中切换到一个左右双栏分布的复杂模板。
  2. 导出为 DOCX。
  3. 在 Office Word 中打开，发现原本的双栏变成了单栏的流式排版，且存在局部样式错位。
* **影响**：`packages/docx/src/builder.ts` 是基于 HTML to DOCX 进行的流式内容构建。求职者下载 DOCX 往往为了微调格式，如果保真度过低，将打击用户导出此格式的积极性。
* **建议修复位置**：在 DOCX 导出操作点（例如 `apps/web/src/routes/dashboard/resumes/index.tsx` 附近）增加产品提示词：“复杂模板导出的 DOCX 为流式纯文本结构，如需完美保留排版样式，请使用 PDF 导出”。

#### 6. OCR Provider 渠道单一且局限于 Azure
* **复现步骤**：
  1. 进入设置 -> AI/OCR Providers 页面。
  2. 发现 OCR 引擎选项仅支持通过 Endpoint 和 API Key 配置（底层硬编码适配 Azure Document Intelligence）。
* **影响**：对于国内个人开发者和求职者来说，注册并绑定外币卡开通 Azure 服务的门槛较高。
* **建议修复位置**：在后端 `packages/api/src/features/ai/ocr.ts` 以及前端的设置表单中，后续逐步扩展接入国内常见的 OCR 引擎（如阿里云、百度 AI），或直接兼容使用支持多模态的 LLM 直读图片。
